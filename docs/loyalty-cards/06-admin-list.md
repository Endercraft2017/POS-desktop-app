# Step 06 — Admin loyalty-card list in Checkout

> A collapsible "Loyalty cards" section in the Checkout page, visible to **admins only** (matches the existing pattern: `currentEmployee?.role === "admin"` gates other sensitive UI).
>
> *Note for Q2 (any-employee stamping):* cashiers can still scan + stamp + claim via the **scanner** button (step 04). They just don't see the full cards list / search — that's admin-only management.

---

## Where it lives

Same Checkout sidebar where the scan button lives. Below the cart, in a collapsible card:

```
+---- Cart sidebar ----+
| Customer: [_______]  |
| [Scan loyalty card]  |
| Payment: [Cash|Card] |
| Total: ₱123.00       |
| [Charge]             |
|                      |
| ▼ Loyalty cards (50) |   ← collapsible header (admin only)
| [🔍 Search code/name] |
| [⬇ Download all]      |   ← NEW per Q5 — zip of all 50 PNGs
| ┌──────────────────┐ |
| │ 47391 Cheng D… 3 │ |
| │ 82610 (unnamed) 0│ |
| │ 11042 Khassy R… 9│ |
| │ …                │ |
| └──────────────────┘ |
| [+ Bulk generate]    |   ← only if 0 cards exist
+----------------------+
```

If the section gets noisy in a cluttered cart sidebar, we can move it to a dedicated `/loyalty-cards` route — but starting here matches the user's requirement ("list them in the loyalty cards list within checkout").

---

## List item layout (per row)

```
[code]  [name or "(unnamed)"]  [stamps/9 progress bar]   [action]
```

- **code**: monospace, small
- **name**: ellipsis-clipped if long
- **stamps**: tiny 9-segment progress bar visualized like the modal grid (3 segments per row × 3 rows)
- **action**: nothing — clicking the whole row opens the LoyaltyCardModal

---

## Sorting / filtering

- **Default sort**: `last_seen_at DESC NULLS LAST, code ASC` (active customers first; brand-new unscanned cards at the bottom)
- **Search**: text input above the list filters by code OR name (case-insensitive substring)
- **No pagination needed** at 50 cards. If the user generates many more (e.g. 500), wrap in a virtualizer like `@tanstack/react-virtual` (already in deps per other pages).

---

## "Download all" button (per Q5)

A button labeled **"⬇ Download all"** sits above the list. Clicking it triggers the browser to download `loyalty-cards.zip` — the same zip produced by the seed script in [step 03](03-qr-generation.md), already deployed to `https://3ks.afkcube.com/app/loyalty-cards.zip` via `deploy-web.bat`.

Simplest implementation:

```tsx
<a
  href="/app/loyalty-cards.zip"
  download="loyalty-cards.zip"
  className="download-all-btn"
>
  ⬇ Download all ({cards.length} PNGs)
</a>
```

Pros: zero new server work, browser handles everything, button always points at the current zip.

**Caveat: the zip is generated at seed-script time, not on demand.** If the admin manually creates additional cards later via the UI (currently out of scope for v1 — admin manages cards by physically reissuing them and re-running the seed script), the zip won't include those. Document this in the button's tooltip:

> "Includes all 50 PNGs generated by the most recent run of the seed script. Re-run `generate-loyalty-cards` if you've added cards since."

If on-demand zipping becomes important later, add a server-side `GET /api/loyalty/cards.zip` endpoint that streams a fresh zip from all current rows. Not v1.

## "Bulk generate" button

Only shown when `listAll()` returns 0 rows. Clicking it:

1. Confirms with a modal: "Generate 50 new loyalty cards? This adds 50 rows to the database and saves 50 QR PNGs to the server."
2. Runs the seed script's logic in-renderer (calling `loyaltyCardRepo.create()` 50 times with random codes) — OR opens a help modal explaining to run the CLI script (see step 03).

*Recommendation: open the help modal.* Bulk generation is a one-time admin task that benefits from being done from the dev machine where the PNGs are produced. Doing it in the renderer would require shipping the QRCode library to the runtime bundle (already there for the modal preview, so this isn't really a blocker, but the PNG-to-server upload step needs the file-system access of the CLI script regardless).

Both paths can coexist; v1 ships the CLI-only path.

## Component sketch

```tsx
// inside CheckoutPage.tsx (admin-only section)
function LoyaltyCardsSection() {
  const { data: cards = [] } = useQuery({
    queryKey: ["loyalty-cards", "all"],
    queryFn: () => loyaltyCardRepo.listAll(),
    refetchOnWindowFocus: false,
    staleTime: 30_000,
  });
  const [search, setSearch] = useState("");
  const [openCardId, setOpenCardId] = useState<string | null>(null);
  const filtered = useMemo(() => {
    const q = search.trim().toLowerCase();
    if (!q) return cards;
    return cards.filter(
      (c) =>
        c.code.includes(q) ||
        (c.customer_name && c.customer_name.toLowerCase().includes(q)),
    );
  }, [cards, search]);

  return (
    <CollapsibleSection title={`Loyalty cards (${cards.length})`}>
      <input
        placeholder="Search code or name…"
        value={search}
        onChange={(e) => setSearch(e.target.value)}
      />
      <ul>
        {filtered.map((c) => (
          <li key={c.id} onClick={() => setOpenCardId(c.id)}>
            <code>{c.code}</code>
            <span>{c.customer_name || "(unnamed)"}</span>
            <StampMiniBar stamps={c.stamps} claimedMask={c.rewards_claimed_mask} />
          </li>
        ))}
      </ul>
      {cards.length === 0 && <BulkGenerateHint />}
      {openCardId && (
        <LoyaltyCardModal cardId={openCardId} onClose={() => setOpenCardId(null)} />
      )}
    </CollapsibleSection>
  );
}
```

`StampMiniBar` is a 3×3 dot visual; reuses the same color tokens as the modal grid.
